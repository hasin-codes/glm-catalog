## 🤔 What is GLM Catalog?

Before contributing, it's important to understand what this project is — and what it isn't.

**GLM Catalog is a catalog of skills and rules** that developers use to steer and instruct Z.ai's GLM model more effectively.

- **Skills** are specialized instruction sets that tell GLM how to handle specific tasks (e.g., generating Express routes, running security audits, creating architecture diagrams)
- **Rules** are instruction files that shape GLM's overall behavior — defining coding conventions, framework preferences, and best practices

**GLM Catalog is NOT:**
- A social platform or forum
- A community discussion board
- A place to post articles or blog posts

**GLM Catalog IS:**
- A searchable, browsable catalog of instruction sets for GLM
- A resource that helps developers get better results from GLM immediately
- A place where the best ways to instruct GLM are curated and shared

When contributing, keep this focus in mind. Every addition should make it easier for a developer to find and use an instruction set that improves their GLM workflow.


```bash
npm run build
```

Ensure the production build completes without errors.

### Keeping Your Fork in Sync

```bash
git fetch upstream
```

## 🏗 Project Architecture

Understanding the project structure will help you contribute effectively:

```
glm-catalog/
├── app/              → Next.js App Router (pages & layouts)
├── components/       → React components (UI + feature components)
│   └── ui/           → Shadcn UI primitives
├── data/             → Static data layer (skills, rules, events, etc.)
│   ├── skills/       → Skill SKILL.md files (source of truth)
│   └── rules/        → Rule .md files (source of truth)
├── lib/              → Utility functions
├── scripts/          → Build scripts (e.g., generate-skills.ts)
└── public/           → Static assets (images, icons)
```

### Key Concepts

1. **Static Data Layer**: All content is stored as TypeScript constants or Markdown files — no database required
2. **Auto-Generated Data**: `data/skills.ts` is generated from `data/skills/**/*.md` via `scripts/generate-skills.ts`
3. **Component-Driven UI**: Each page is composed of reusable components
4. **Dark Theme**: The entire app uses a dark color scheme (`#0a0a0a` background, `#111` cards)
5. **Shadcn UI**: UI primitives live in `components/ui/` and are managed via the Shadcn CLI

## 🧠 Adding New Skills

Skills are the most important content in GLM Catalog. A **skill** is a specialized instruction set that tells GLM exactly how to approach a specific task. When a developer copies your skill into their GLM configuration, GLM will follow those instructions whenever it encounters that type of task.

### What Makes a Good Skill?

A good skill:
- **Solves a specific problem** — e.g., "generate Express routes" not "help with backend"
- **Is actionable** — provides clear, step-by-step instructions GLM can follow
- **Includes examples** — shows GLM what good output looks like
- **Defines boundaries** — specifies what tools GLM can use and what it shouldn't do
- **Is well-tagged** — makes it easy to discover through search and category browsing

### Step 1: Choose a Category

Browse existing categories in `data/skills/`:

```
data/skills/
├── api-development/          # REST, GraphQL, OpenAPI
├── api-integration/          # Webhooks, OAuth, SDKs
├── aws-skills/               # Lambda, S3, EC2
├── backend-dev/              # Express, FastAPI, Django, NestJS
├── business-automation/      # Workflow, Zapier, n8n
├── data-analytics/           # SQL, BI, visualization
├── data-pipelines/           # ETL, streaming, warehousing
├── devops/                   # Docker, K8s, CI/CD
├── frontend-dev/             # React, Vue, Angular, Svelte
├── gcp-skills/               # Cloud Monitoring, Firestore, Vertex AI
├── ml-training/              # TensorFlow, MLflow, W&B
├── mobile-dev/               # React Native, Flutter, Swift
├── security-advanced/        # SOC2, PCI-DSS, Zero Trust
├── security-fundamentals/    # JWT, CSRF, XSS, SQL injection
├── technical-docs/           # README, API docs, ADRs
├── test-automation/          # Jest, Vitest, Pytest
├── visual-content/           # Mermaid, PlantUML, charts
└── ...
```

**If your skill doesn't fit an existing category**, create a new one (see [Creating a New Category](#creating-a-new-category)).

### Step 2: Create the Skill File

Create a new file at `data/skills/<category>/<skill-name>/SKILL.md`:

```markdown
---
name: "my-awesome-skill"
description: |
  A brief description of what this skill does. Include trigger phrases
  and context about when this skill should activate. This helps developers
  understand what the skill does before they copy it.
allowed-tools: "Read, Write, Edit, Bash(cmd:*), Grep"
version: 1.0.0
license: MIT
author: "Your Name <your@email.com>"
---

# My Awesome Skill

## Overview

A clear explanation of what this skill provides and why it's useful.
Explain what problem it solves for developers using GLM.

## When to Use

This skill activates automatically when you:
- Mention "my awesome skill" in your request
- Ask about related patterns or best practices
- Need help with [specific domain]

## Instructions

1. Step-by-step guidance for the skill
2. Follow industry best practices
3. Generate production-ready code
4. Validate outputs against common standards

## Examples

**Example: Basic Usage**
Request: "Help me with my awesome skill"
Result: Provides step-by-step guidance and generates appropriate output

**Example: Advanced Usage**
Request: "Configure my awesome skill for production"
Result: Provides production-ready configuration with best practices

## Prerequisites

- List any prerequisites needed
- Tools, libraries, or knowledge required

## Notes

- Additional tips, caveats, or important information
- Links to relevant documentation
```

### Step 3: Regenerate Skills Data

After creating your skill file, regenerate the TypeScript data:

```bash
npx tsx scripts/generate-skills.ts
```

This will update `data/skills.ts` with your new skill, including proper typing and categorization.

### Step 4: Verify

```bash
npm run dev
```

Navigate to `/skills` and verify your skill appears in the correct category. Click through to the detail page to ensure everything renders correctly. Test the copy button to make sure the content copies properly.

### Skill File Reference

| Field | Required | Description |
|-------|----------|-------------|
| `name` | ✅ | Kebab-case identifier (e.g., `express-route-generator`) |
| `description` | ✅ | 1-3 sentence description with trigger phrases. Helps developers decide if this skill is what they need. |
| `allowed-tools` | ✅ | Comma-separated list of tools the skill instructs GLM to use |
| `version` | ✅ | Semantic version (e.g., `1.0.0`) |
| `license` | ✅ | License identifier (e.g., `MIT`, `Apache-2.0`) |
| `author` | ✅ | Author name and optional email |

### Creating a New Category

If your skill doesn't fit any existing category:

1. Create the directory: `data/skills/<new-category>/`
2. Create a `README.md` inside it:

```markdown
# Category Name

A brief description of this skill category and what types of tasks
the skills in this category help GLM handle.

## Tags

`tag1`, `tag2`, `tag3`
```

3. Add your skill inside the new category directory
4. Run `npx tsx scripts/generate-skills.ts`

## 📜 Adding New Rules

**Rules** are instruction files that steer GLM's overall behavior — they define coding conventions, framework preferences, and best practices that GLM follows across all its output. Unlike skills (which activate for specific tasks), rules are always active and shape GLM's general approach.

### What Makes a Good Rule?

A good rule:
- **Is framework-specific** — targets a particular language, framework, or tool
- **Enforces conventions** — tells GLM how to name things, structure code, format output
- **Is practical** — covers real-world patterns developers actually use
- **Is concise** — long enough to be useful, short enough to be readable

### Step 1: Check Existing Rules

Browse `data/rules/` to see existing categories and avoid duplicates.

### Step 2: Create the Rule File

Add your rule as a `.md` file in the appropriate category directory:

```markdown
---
title: "React Best Practices"
author:
  name: "Your Name"
  avatar: "https://avatars.githubusercontent.com/u/XXXXX"
tags:
  - react
  - best-practices
  - frontend
---

# React Best Practices

Your rule content goes here. This is the actual text that will be
copied by developers and added to their GLM configuration.

- Use functional components with TypeScript
- Prefer named exports over default exports
- Use Tailwind CSS for styling
- Implement proper error boundaries
- Use React.memo for expensive renders
- Follow the RORO (Receive Object, Return Object) pattern
```

### Step 3: Verify

Run `npm run dev` and navigate to `/rules` to see your new rule appear. Test the copy button.

### Rule Guidelines

- **Be specific**: Rules should target a particular framework or language
- **Be actionable**: Every line should tell GLM something concrete to do
- **Be concise**: Aim for rules that are useful at a glance
- **Tag properly**: Use relevant tags for discoverability
- **No duplicates**: Search existing rules before adding new ones

## 🎨 UI / Frontend Contributions

### Adding Shadcn UI Components

```bash
npx shadcn@latest add <component-name>
```

This installs the component into `components/ui/` with proper TypeScript types and Tailwind styling.

### Component Conventions

- **Functional components** using `function` keyword (not arrow functions)
- **TypeScript interfaces** for props (not `type` aliases)
- **Named exports** for all components
- **File structure**: Exported component → subcomponents → helpers → static content → types
- **Minimize `"use client"`**: Only use for Web API access, event handlers, or hooks
- **Dark theme only**: All colors should work on `#0a0a0a` background

### Color Palette

| Usage | Color | Tailwind Class |
|-------|-------|---------------|
| Background | `#0a0a0a` | `bg-[#0a0a0a]` |
| Card Background | `#111111` | `bg-[#111]` |
| Primary Text | `#ffffff` | `text-white` |
| Secondary Text | `#888888` | `text-[#888]` |
| Muted Text | `#555555` | `text-[#555]` |
| Border Default | `rgba(255,255,255,0.1)` | `border-white/10` |
| Border Hover | `rgba(255,255,255,0.2)` | `border-white/20` |

### Responsive Design

- **Mobile-first** approach using Tailwind breakpoints
- Default styles target mobile, then layer `sm:`, `md:`, `lg:` modifiers
- Test at 320px, 768px, 1024px, and 1440px widths

## 📏 Coding Standards

### TypeScript

```typescript
// ✅ DO: Use interfaces
interface SkillCardProps {
  skill: Skill
  isPage?: boolean
}

// ❌ DON'T: Use type aliases for object shapes
type SkillCardProps = {
  skill: Skill
  isPage?: boolean
}

// ✅ DO: Use function keyword for components
function SkillCard({ skill, isPage }: SkillCardProps) {
  return <div>...</div>
}

// ❌ DON'T: Use arrow functions for components
const SkillCard = ({ skill, isPage }: SkillCardProps) => {
  return <div>...</div>
}

// ✅ DO: Use descriptive variable names
const isLoading = true
const hasFetched = false

// ❌ DON'T: Use abbreviated names
const loading = true
const fetched = false
```

### Imports

```typescript
// ✅ DO: Group imports logically
import type { Metadata } from "next"
import { Geist, Geist_Mono } from "next/font/google"
import { Toaster } from "sonner"

import Navbar from "@/components/Navbar"
import Footer from "@/components/Footer"
import { cn } from "@/lib/utils"

// ✅ DO: Use @/ path alias for project imports
import { Skill } from "@/data/skills"

// ❌ DON'T: Use relative paths for cross-directory imports
import { Skill } from "../../data/skills"
```

### Styling

```typescript
// ✅ DO: Use Tailwind utility classes
<div className="rounded-xl border border-white/10 bg-[#111] p-5">

// ✅ DO: Use cn() for conditional classes
<div className={cn(
  "rounded-xl border bg-[#111]",
  isActive && "border-white/20",
  isPage && "max-w-4xl mx-auto"
)}>

// ❌ DON'T: Use inline styles
<div style={{ borderRadius: '12px', border: '1px solid rgba(255,255,255,0.1)' }}>
```

### Error Handling

```typescript
// ✅ DO: Handle errors early with guard clauses
function processSkill(skill: Skill | null) {
  if (!skill) return null
  if (!skill.content) return null

  // Happy path
  return transformSkill(skill)
}

// ❌ DON'T: Deeply nest error handling
function processSkill(skill: Skill | null) {
  if (skill) {
    if (skill.content) {
      return transformSkill(skill)
    }
  }
}
```

## 📝 Commit Messages

We follow [Conventional Commits](https://www.conventionalcommits.org/) for clear, structured commit history:

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

### Types

| Type | Description | Example |
|------|-------------|---------|
| `feat` | New feature | `feat(skills): add kubernetes skill category` |
| `fix` | Bug fix | `fix(rules): resolve sidebar scroll overflow` |
| `docs` | Documentation | `docs: update contributing guide` |
| `style` | Formatting | `style: fix linting errors in SkillCard` |
| `refactor` | Code restructuring | `refactor(data): simplify skills parser` |
| `perf` | Performance | `perf: lazy-load skill grid components` |
| `test` | Tests | `test: add unit tests for utils` |
| `chore` | Maintenance | `chore: update dependencies` |

### Examples

```bash
feat(skills): add graphql-resolver-creator skill
fix(navbar): resolve mobile menu not closing on navigation
docs(readme): add skills system documentation
style(components): consistent border-radius across cards
refactor(scripts): improve SKILL.md frontmatter parser
```

## 🔄 Pull Request Process

### Before Submitting

1. **Sync with upstream**: Make sure your branch is up to date with `main`
2. **Run linter**: `npm run lint` — fix all errors
3. **Build check**: `npm run build` — ensure no build errors
4. **Test locally**: `npm run dev` — verify your changes work
5. **Regenerate data**: If you added/modified skills, run `npx tsx scripts/generate-skills.ts`

### PR Title Format

Use the same Conventional Commits format:

```
feat(skills): add new kubernetes-rbac-analyzer skill
fix(ui): resolve mobile layout overflow on rules page
docs: update README with new features
```

### PR Description Template

```markdown
## Summary

Brief description of what this PR adds or changes and how it helps developers use GLM better.

## Changes

- [ ] Added new skill: `skill-name` — helps GLM handle [specific task]
- [ ] Fixed bug in `component-name`
- [ ] Updated documentation

## Testing

- [ ] Tested on Chrome (desktop)
- [ ] Tested on mobile viewport
- [ ] Ran `npm run lint` — no errors
- [ ] Ran `npm run build` — successful
- [ ] Verified skill/rule appears correctly on catalog pages
- [ ] Tested copy button functionality

## Related Issues

Closes #123
```

### Review Process

1. A maintainer will review your PR within a few days
2. They may request changes — respond promptly and push updates
3. Once approved, your PR will be merged into `main`
4. Your skill or rule is now helping developers use GLM better! 🎉
